- **T3.3** done 2026-07-12. Dependency resolution: `resolveDependencies(store, bundlesDir, framework)` -- a generic graph resolver over `dependencies.settings`/`dependencies.roles`/`dependencies.plugins` (§6.3) that knows nothing about specific capability or plugin-type names, only the declared dependency shape. Loads manifests for every currently role-bound trusted plugin from `wanfw_bundles`, checks each declared setting against the framework doc, each declared role against current role bindings, each declared plugin dependency against the trust store, and runs a DFS cycle check over the plugin-dependency graph among role-bound plugins. The load-bearing fact about this API (undocumented in most client libraries but critical here): `setHosts` **replaces the entire host record set** for a domain in one call -- there is no incremental add/delete endpoint -- so `apply.ts`'s `applyDnsRecord` (the `dns.apply` task, to be brokered by T4.3) always does a full `getHosts -> merge in memory by (type, name) -> setHosts` round trip; `action: "set"` upserts, `action: "delete"` filters out, and a delete of a never-existing record is a harmless no-op (load-bearing for T4.4's "always attempt cleanup, even on failure paths" requirement). A hand-rolled regex-based XML reader handles Namecheap's fixed, well-known response shape (flat self-closing `<host .../>` tags, `<Errors><Error Number="...">`) -- a full XML parser dependency would solve a much bigger problem than this API poses. §9's specific requirement ("detect 403s and say add this host's WAN IP to the Namecheap API allowlist") is implemented as a small set of known IP-allowlist error codes that get a distinct, actionable message; every other API error surfaces its raw Namecheap message unchanged.

  **A genuinely new, non-obvious infrastructure bug found only by live verification, worth its own paragraph:** the plugin initially used the global `fetch()` API (matching the rest of the codebase's Node-native style) for both the WAN-IP lookup and the Namecheap calls. Every invocation failed in the real Docker stack with an opaque `error: "connection closed"` and zero JS-level stderr output -- the classic signature of a native (non-JS-catchable) crash. Root cause, found by temporarily piping the child's stderr (previously silently discarded, `packages/pluginhost/src/child-runner.ts`) into the pluginhost's own log stream: a native V8 assertion failure on the very first `fetch()` call, then (after raising the `--as` ceiling to see past it) a clean but telling `RangeError: WebAssembly.instantiate(): Out of memory: Cannot allocate Wasm memory for new instance` from `node:internal/deps/undici`. Node's global `fetch()` is implemented via `undici`, whose HTTP parser (`llhttp`) is WASM-compiled; WASM engines commonly reserve several GB of *virtual* address space up front for guard-page-based bounds checking, entirely unrelated to actual memory used -- and this reservation is fundamentally incompatible with `prlimit --as` (ADR-3's per-invocation sandbox mechanism, which caps virtual address space, not resident memory). Confirmed by raising the invocation's `memMb` ceiling all the way to 8192 and still hitting the identical WASM OOM -- proving this isn't a "give it more memory" problem, it's a wrong-dimension-of-limit problem that no reasonable `--as` value fixes. The real fix: switch to `node:https` directly (the classic, non-WASM HTTP client) for every outbound call this plugin makes, which resolves cleanly within the *original* T2.9 768MB floor with zero ceiling changes needed anywhere. Documented directly in `main.ts` as the load-bearing reason, not a style note, since **every future plugin needing outbound HTTP under this sandbox has the same landmine** (T4.4's cert-letsencrypt-dns01 will need this same discipline, noted ahead of time for that task). The stderr-piping fix in `child-runner.ts` is kept as a permanent, generally useful diagnostic (previously *any* plugin crash of this class would have been invisible; now it's captured and prefixed per-plugin in the pluginhost's own logs) -- confirmed via `pluginhost` unit tests still green afterward, unaffected by the addition.

  **The harness itself:** `deploy/docker-compose.pebble.yml` adds `pebble` (Let's Encrypt's own ACME v2 test server, pulled from `ghcr.io/letsencrypt/pebble` -- the `letsencrypt/pebble` Docker Hub name used in most public examples no longer resolves) and `pebble-challtestsrv` (its companion fake-DNS + management API), pointing `pluginhost` at them via `WANFW_ACME_DIRECTORY_URL`/`WANFW_DNS01_RESOLVER`/`WANFW_CHALLTESTSRV_URL`/`NODE_TLS_REJECT_UNAUTHORIZED`. `plugins/dns-mock` (new, T4.7) is a Pebble-only `dns-provider` plugin whose `dns.apply` calls challtestsrv's `/set-txt`/`/clear-txt` HTTP API instead of a real DNS API -- exactly what Pebble's own DNS-01 validator queries directly (via its `-dnsserver` flag), so the whole flow needs no real domain or DNS ownership anywhere. `test/integration/pebble-e2e.sh` (new) drives it end to end: trust builtins, write a framework+service doc binding `dnsProvider: dns-mock` / `certIssuer: cert-letsencrypt-dns01`, wait for the RENEWAL stage to auto-issue, assert the stored cert is a real Pebble-signed PEM, assert the TXT record was cleaned up, force a renewal by backdating `storedAt` past the 30-day window and clearing `renewal-state.json`, assert a real second issuance lands as generation 2, assert the live `Caddyfile` in `wanfw_proxycfg` references the new generation's real paths. Runs in well under a minute, tears the whole stack down after (same `docker compose down -v` discipline as `run.sh`/`m1-plugin-runtime.sh`).

  **Bug 1 -- child env was always fully stripped, silently.** `child-runner.ts` spawns every plugin invocation with a clean env (`{PATH: "/usr/bin:/bin"}` only, by design -- ADR-3, no ambient host config leaking to children). This meant `WANFW_ACME_DIRECTORY_URL` etc. set on the pluginhost *container* never reached the spawned child at all -- a manual `cert.ensure` invocation pointed "at Pebble" was silently hitting real production Let's Encrypt instead, and the real-LE policy-block error it returned for a throwaway test domain looked enough like a plausible error that it took a second, unrelated-domain test to notice production was being hit at all. Fixed with a small explicit non-secret allowlist (`ENV_PASSTHROUGH_ALLOWLIST`: the four ACME/DNS endpoint-override vars above, never anything credential-shaped) forwarded from the pluginhost process's own env into every child's -- empty in the real production compose file, so this is a no-op there. New `childEnv()` unit tests (`child-runner.test.ts`): empty by default, forwards exactly the allowlisted vars set, never forwards an arbitrary non-allowlisted var.

  **Bug 2 -- `node:http` (not `https`) crashes under this sandbox, but only from ESM, and it's a different bug from T4.2's.** `dns-mock`'s plain `node:http.request` to challtestsrv reliably crashed the spawned child with the exact `WebAssembly.instantiate(): Out of memory` error T4.2 documented for `fetch()` -- but `dns-mock` never calls `fetch()`, only `node:http`. Isolated by direct experimentation (a minimal CJS `-e` script using the identical `http.request` call never crashed; the identical logic saved as a real `.mjs` file and run as ESM crashed every time; `node:https` in ESM never crashed) down to "`node:http` (not `https`) + ESM `import` (not CJS `require`)" -- confirmed unrelated to the memMb ceiling itself, same as T4.2's finding (raising it to 2048MB made no difference). Fixed by rewriting `dns-mock`'s client as a hand-rolled minimal HTTP/1.1 POST over a raw `node:net` socket, which never touches Node's HTTP client module at all -- same "hand-roll it, sidestep the whole broken subsystem" discipline this codebase already applies to XML parsing (T4.2) and ASN.1 DER encoding (T4.4). Documented in `main.ts`'s own comment as a landmine for any future plugin talking to a plain-HTTP (not HTTPS) backend.

  **Bug 3 -- authorization reuse silently broke every second issuance for the same name, i.e. every real renewal.** RFC 8555 §7.1.4 lets a server reuse an already-`"valid"` authorization for an identifier instead of issuing a fresh pending one; Pebble does this probabilistically (its own boot log: "attempt authz reuse ... 50% of the time") and so does real production Let's Encrypt. `cert-ensure.ts` unconditionally POSTed a challenge response regardless of the fetched authorization's status -- a protocol violation against an already-valid authorization, which Pebble was entitled to reject, and did, with no explicit error surfacing anywhere in this session's logging (the RENEWAL stage's own failure path is deliberately silent-but-for-`consecutiveFailures`, T4.6's own design). This is the T4.7 finding most likely to have caused real, silent production breakage: the very first time Let's Encrypt happened to reuse an authorization on a routine renewal, that renewal would have failed forever, every day, with nothing visible short of watching `wanfw cert list`'s generation number stop advancing. Fixed: skip an already-`"valid"` authorization entirely (no TXT record needed at all for it), covered by a new `cert-ensure.test.ts` case using a `fakeAcmeServer({authStatus: "already-valid"})` variant.

  **Bug 4 -- the proxy's config volume could get silently, permanently poisoned by Docker's own volume-seeding behavior.** Two compounding issues, both found only once the proxy container was actually mounting the *real* project-prefixed volumes for the first time (see next paragraph) and its config still didn't reflect reality: (a) `proxy-container.ts` hardcoded the literal volume names `wanfw_certs`/`wanfw_proxycfg`, but Docker Compose always names a project's volumes `<project>_<key>` -- under this repo's `docker-compose.yml` (`name: wanfw`, volume keys `wanfw_certs`/`wanfw_proxycfg`), the volumes that actually exist are `wanfw_wanfw_certs`/`wanfw_wanfw_proxycfg`. The mismatch never errored -- the Docker API silently auto-creates a new, empty, phantom volume under an unrecognized name instead of failing -- so the proxy container has been running since T3.12 quietly serving Caddy's own built-in sample config from a volume nothing else ever wrote to, and no prior live-verification session's checks (container exists, Caddy logs "loaded successfully") were deep enough to notice, since a trivial default Caddyfile also "loads successfully." Fixed: `buildProxyContainerSpec` now takes the real volume names as parameters (`ExecuteStageDeps.certsVolumeName`/`proxycfgVolumeName`, sourced from new `paths.ts` fields `WANFW_CERTS_VOLUME_NAME`/`WANFW_PROXYCFG_VOLUME_NAME`, set correctly on the orchestrator service in `docker-compose.yml`). (b) Once mounting the *correct* volume, a second, independent bug surfaced: Docker seeds a genuinely empty named volume from whichever container's image has content at that mount path on that container's first-ever creation -- if the proxy (`caddy:2`) container happened to be created before the orchestrator ever wrote a real file into `proxycfgDir` (EXECUTE creates the proxy container *before* its first config write/reload, since reload needs a running container to `docker exec` into), the volume got seeded with Caddy's own default sample config, owned by Caddy's uid, and every later orchestrator write failed with `EACCES` forever after. Fixed: `main.ts` now writes a real placeholder `Caddyfile` at boot, unconditionally before any reconcile could possibly run, guaranteeing the orchestrator is always the first real writer regardless of container creation order; EXECUTE's first real pass overwrites it with the actual rendered config. A closely related, much smaller Caddyfile-syntax bug found in the same debugging pass: `proxy-caddy`'s catch-all was a single `:443, :80 { tls ... }` block, which Caddy correctly rejects (`server listening on [:80] is HTTP, but attempts to configure TLS connection policies`) -- split into two separate blocks, `:443` (with `tls`) and `:80` (without), fixed with 2 new/updated `proxy-caddy` render tests.

- **T5.1** done 2026-07-12. IPAM host API (ADR-1, §6.6) -- the first task of Phase 5 (M4), and the piece `network-macvlan` (T5.2) needs before it can hand out a static proxy IP.

  `packages/orchestrator/src/ipam/cidr.ts` is pure IPv4 CIDR math with zero dependencies: `parseCidr` (network/broadcast integers from a `"a.b.c.d/n"` string), `hostsInCidr` (every usable host address in order, excluding the network address, the broadcast address, and an optional extra exclusion -- the range's own gateway, always passed for a macvlan range), `isIpInCidr`. IPv4-only and deliberately unoptimized for anything past a typical LAN-sized slice (a macvlan reserved range is always small, per §8.4's own framing -- "a reserved CIDR slice outside the DHCP pool"), so "enumerate every host and find the first free one" is the whole allocation algorithm, no bitmap or interval-tree needed.

  `StateStore` gained `setIpamRange`/`getIpamRange` (upsert, keyed by a short id like `"macvlan"` -- one row per network-provider that needs address bookkeeping, not per-plugin) and `allocateIp`/`releaseIp`/`listIpamAllocations`, backed by the `ipam_ranges`/`ipam_allocations` tables T2.1 had already scaffolded in `migrations.ts` but never implemented. `allocateIp` runs inside a transaction (finds the first host in the range's CIDR with no *live* allocation row, inserts) so two concurrent allocate calls can never race onto the same address; exhaustion throws a new `IpamExhaustedError`, a distinct type specifically so the host-API dispatcher can tell "no capacity" apart from "no such range" apart from a real bug, and respond to each differently. `releaseIp` is a soft-release (upsert on re-allocation of the same address, not a fresh row) -- idempotent by design (double-release is a no-op, matching §9's general "cleanup is idempotent" discipline elsewhere in this codebase), and a released address is immediately eligible for reuse, first-in-CIDR-order same as a never-allocated one.

  Two new capability-gated -- well, *structurally* gated, not capability-gated -- host API methods: `ipam.allocate`/`ipam.release`. The plan's own wording is "implicit for `network-provider` plugins," which this implements literally rather than via a capability grant: `callingPluginTypes()` reads the calling plugin's own currently-trusted manifest and checks `types.includes("network-provider")`, the same "structural check, no grant lookup" category `state.get/put` already established (T4.1's own dispatcher docstring) -- IPAM bookkeeping isn't a powerful/dangerous action in the sense secrets or DNS writes are, so gating it behind an explicit operator-approved grant would be spec-inaccurate ceremony. Reused `dependency-resolution/resolve.ts`'s own `readManifest(bundlesDir, sha256)` (exported for this, rather than duplicating a second manifest reader -- caught by a real `tsc` "ambiguous export" error when a first attempt at this accidentally created a duplicate in `composed-schema/compose.ts` instead).

  The range itself is synced from desired state, not passed in by the caller: `buildLoadStage` (`core-stages.ts`) now also upserts an `id: "macvlan"` ipam range from `framework.spec.network.macvlan.{reservedCidr,gateway}` on every load, exactly the same "sync a mutable holder from desired state on load" shape T4.3 already established for `FrameworkRolesHolder` -- a changed CIDR/gateway in the framework doc takes effect on the very next reconcile, no separate stage or event needed. No macvlan config configured at all is a silent no-op (bridge-only deployments never touch the ipam tables).

  `probe()` is deliberately thin: decline immediately if no `defaultRouteInterface` was ever supplied in `ProbeContext` (nothing to attempt macvlan against -- detection itself is the wizard's job, T5.3, not this plugin's; ADR-1's own design-spec text says "wizard probe auto-detects the default-route interface," so `ProbeContext.defaultRouteInterface` is an *input*, not something this plugin computes), otherwise defer entirely to a new core-mediated host API method, `net.probeNetwork`, and pass its `ok`/`reason` straight through unchanged -- this plugin has no opinion of its own about *why* macvlan would fail on a given host (MAC filtering, no promiscuous mode, a WiFi uplink), so re-interpreting the daemon's real error would only throw away information an operator needs.

The wizard asks the operator directly instead (`ip route` on the host names it in one line) -- honest about the actual constraint rather than a half-built detector.

  **A real, live-triggered readline bug, found only by actually running the wizard through its real invocation path (`docker exec -i`, no TTY) rather than only the injected-prompt unit tests:** the first working version used `node:readline/promises`, one fresh `Interface` per question. It failed on the very first live run -- later questions silently returned empty strings, an outcome none of the (correctly passing) unit tests caught, since they inject `prompt()` directly and never exercise real stdin at all. Root cause: `docker exec -i` delivers piped stdin in one burst with no TTY, and Node's readline fires `'line'` events as data arrives regardless of whether a `.question()` call is actively awaiting one -- lines arriving between two `.question()` calls (which, over piped input, is *every* line) can be delivered to the wrong call or dropped.
