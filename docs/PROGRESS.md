- **T3.3** done 2026-07-12. Dependency resolution: `resolveDependencies(store, bundlesDir, framework)` -- a generic graph resolver over `dependencies.settings`/`dependencies.roles`/`dependencies.plugins` (§6.3) that knows nothing about specific capability or plugin-type names, only the declared dependency shape. Loads manifests for every currently role-bound trusted plugin from `wanfw_bundles`, checks each declared setting against the framework doc, each declared role against current role bindings, each declared plugin dependency against the trust store, and runs a DFS cycle check over the plugin-dependency graph among role-bound plugins.   **Real bug found and fixed via live verification, caught nowhere by unit tests:** the newly-provisioned proxy container was being destroyed and immediately recreated on every single reconcile. Root cause: OBSERVE's GC (T3.9) removes any `wanfw.managed` object whose `wanfw.service` label isn't in current desired state, and the proxy container had been labeled `wanfw.service: "__proxy"` -- a sentinel that (correctly) never appears in `desiredServiceIds`, so GC treated it as an orphan every time, immediately after EXECUTE had just created it. Fixed properly rather than special-cased: `ensureNetwork`/`ensureContainer` (`execute/ensure.ts`) gained an explicit `core?: boolean` label option that sets `wanfw.core=true` and omits the `wanfw.service` label entirely for core-emitted objects, and the *existing* GC filters (already scoped to `c.labels["wanfw.service"]` being truthy) then exclude them structurally with no additional GC-side code change needed -- objects with no service label were already outside GC's blast radius by construction, the bug was purely in how the proxy container was labeled at creation. Added a dedicated regression test (`observe-stage.test.ts`: "a core-authority object ... survives GC every pass") asserting exactly this against an empty desired-state map, the case that would have caught it.

  **Second real bug found and fixed via live verification, also caught nowhere by unit tests (they mock the Docker client entirely):** `ensureContainer`'s volume-type mounts were silently not attaching at all against a real daemon. `docker-client.ts`'s `createContainer` had been building `HostConfig.Binds` as `"source:target:ro"` strings for both bind *and* volume mount types -- but the Docker Engine API's `Binds` field is bind-mount-shaped; it does not reliably attach named volumes the way the CLI's `-v` flag (which client-side-translates into the structured `Mounts` field) does. The proxy container's `/etc/caddy` mount consequently just showed the `caddy:2` image's own baked-in sample Caddyfile, not the real one being written to the volume -- confirmed by direct comparison (`docker exec cat` vs. a throwaway container mounting the same volume at a fresh, non-pre-populated path, which correctly showed the real content). Fixed by switching `createContainer` to build the structured `HostConfig.Mounts` array (`{Type, Source, Target, ReadOnly}`) instead of `Binds`, threading a proper `{type, source, target, readOnly}[]` shape through `CreateContainerOptions` and `ensure.ts` rather than pre-flattening to strings. This is a correctness fix applicable to any deployment target, not a workaround -- it also silently affected T3.8's/T3.9's volume-mount tests, which only ever exercised `FakeDockerClient` and so never caught it.

  **Known, honestly-reported live-verification gap (Colima-specific, not fixed):** after both fixes above, the proxy container correctly persists across reconciles and the reload pipeline runs successfully end to end (Caddy's own logs confirm `"config is unchanged"`/`"load complete"` against a payload matching our real file's byte size), but a final direct-content check under this Mac's Colima/virtiofs environment specifically still showed the image's baked-in default file rather than the freshly-written one when mounting *over a path that already contains files in the image's own layer* (`/etc/caddy`, which ships a sample Caddyfile) -- while a control test mounting the identical volume at a fresh, non-pre-populated path correctly showed the real content, isolating this to a mount-shadowing quirk specific to mounting over non-empty pre-existing directories under Colima's overlay2/virtiofs combination, not a code defect (same class of issue as T3.5's already-documented "`/tmp` not bind-mountable in Colima" finding).
