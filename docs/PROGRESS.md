Two real bugs found and fixed only by testing in an actual Linux container (not caught by macOS-only testing, confirming the user's heads-up that this platform mismatch matters): (1) spawning with a fully empty `env` breaks `execvp`-based resolution of both `node` (fixed by using `process.execPath`, an absolute path) and `prlimit` itself (fixed by keeping a minimal `PATH=/usr/bin:/bin`, still "clean" re: secrets/ambient config); (2) an unhandled `EPIPE`/reader `"error"` event when writing to or reading from a child that already died crashed the whole process as an uncaught exception -- fixed with safety-net listeners in `JsonRpcConnection`.   **Deviation:** the spec says the CLI should "confirm interactively" before trust/untrust. Implemented as a two-step `--yes` gate instead of a live y/N readline prompt: without `--yes` the command prints exactly what would happen (manifest id, hash, capabilities) and exits 0 without mutating anything; re-running with `--yes` performs it. This keeps the same "operator must see it and deliberately re-invoke" property ADR-6 wants, while staying scriptable for tests and CI, and avoids the complexity of stdin line-editing in a `docker exec -i` pipe. Revisit if a real interactive prompt turns out to matter for the operator UX.

  Two real bugs found only by running this live, not by unit tests:
  1. **`insertTrustRecord` couldn't re-trust after untrust.** Untrust correctly preserves the row (`revoked_at` set, not deleted) for audit history, but the trust flow did a raw `INSERT`, so re-trusting the same `(plugin_id, version)` hit the `UNIQUE` constraint. Fixed with `ON CONFLICT ... DO UPDATE ... revoked_at = NULL` (upsert); added a store-level regression test for exactly this sequence.
  2. **Default `memMb` for CLI/admin-socket invokes was 256**, below the ~500-700MB floor V8 needs just to start under Linux `prlimit --as` (per the T2.6 finding) — so every `wanfwctl plugin invoke` failed with a misleading `"connection closed"` (the child's stdout ended before it could ever respond, since the process itself never got past V8 init). Raised both defaults to 768MB with a comment pointing at the T2.6 empirical finding.

  Also found and fixed a red herring in my own test script while chasing bug #2: the "tamper" step used `sed 's/hello/HELLO/'` against fixture source that contains no literal "hello" string, so the tamper was silently a no-op — switched to appending a marker line, which reliably changes the hash.

  All four M1 checks verified green against the real stack: (1) trusted echo invoke round-trips input; (2) a tampered `wanfw_bundles` copy is refused with a hash-mismatch error, produces a `plugin.invoke.refused` audit entry, and the audit chain still verifies afterward; (3) a task attempting an out-of-grant host call (`secrets.read` with no grant) gets rejected and the plugin's own error-handling path is exercised; (4) a task that never responds is SIGKILLed at the wall-clock timeout. §12.5 harness reconfirmed green afterward (both suites coexist cleanly in one CI run).
