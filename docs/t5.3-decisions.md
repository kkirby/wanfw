# T5.3 open sub-decisions — resolved (pre-implementation)

The implementation plan (T5.3) deliberately left two architectural sub-decisions
open, requiring both to be "resolved, implemented, and logged in PROGRESS.md."
This document resolves them ahead of implementation; the eventual T5.3
PROGRESS.md entry should fold these in and this file can then be deleted or
left as the long-form record.

Both decisions were made against the *actual* volume ACLs in
`deploy/docker-compose.yml` (verified, not assumed):

| volume | tier1 | orchestrator |
|---|---|---|
| `wanfw_desired` | rw | **ro** |
| `wanfw_status` | **ro** | rw |
| `wanfw_tier1state` | rw (private) | not mounted |

---

## Decision 1: the framework document becomes orchestrator-owned state in `wanfw_state`, authored only via the admin socket

**Decision.** `framework.json` ceases to live in `wanfw_desired`. The framework
document is persisted in the state store (`wanfw_state`, meta table), written
exclusively through a new admin-socket endpoint (`PUT /framework`) that
schema-validates against the same envelope+framework validators the file loader
uses **at write time**, appends a `framework.set` audit entry, and triggers an
immediate reconcile (the established `onApprovalChange`/`onCertChange` callback
pattern). Read back via `GET /framework` (admin) and mirrored **read-only** to
tier1 via a status-socket `GET /framework` route (added to the invariant-#4
route allowlist and its test). Service documents and plugin-config documents
remain files in `wanfw_desired`, authored by tier1, untouched — exactly as the
plan requires.

Hard cutover, not a fallback: LOAD sources the framework doc from the store
only. A leftover `wanfw_desired/framework.json` is *ignored with a logged
warning* (throttled to once per boot), never silently — two sources of truth
with a precedence rule is the trap where someone edits the file and nothing
happens. The integration/e2e scripts (`pebble-e2e.sh` etc.) switch from busybox
volume writes to `wanfwctl framework set` (which they should prefer anyway —
it's less machinery than mounting the volume into a throwaway container).

**Why this option (the plan's own lean, confirmed with sharper reasons):**

1. **The alternative launders away the invariant it claims to preserve.** The
   other candidate — a bootstrap endpoint that stages the doc somewhere tier1
   will copy into `wanfw_desired` — means the orchestrator can author intent
   with extra steps: if tier1 faithfully copies whatever the orchestrator
   stages, the ro mount on `wanfw_desired` protects nothing. It also makes CLI
   init block on another container being up and running a consumer loop. The
   plan called the state-store option "simpler and honest"; honest is the
   load-bearing word.
2. **Authority gets *stronger*, not weaker.** The framework doc is the most
   powerful piece of intent in the system (role bindings, network provider,
   domain). Under this decision it is authorable only via the root-gated admin
   path (admin.sock, 0600, reachable only through docker-exec = host root per
   ADR-6) — not by tier1, the component §12.5 deliberately treats as least
   trusted. tier1 keeps authoring exactly the intent tier that VALIDATE/GATE
   approval-gates downstream: services. This split is principled, not
   accidental: service-level intent from the password-gated LAN UI,
   framework-level intent from host root only.
3. **Write-time validation beats reconcile-time discovery.** This session's own
   live runs hit the file-drop trap twice (envelope-invalid `framework.json`
   discovered only via reconcile-failure log lines). `PUT /framework` fails the
   admin call with the validation message instead.
4. **Audit.** Admin-socket mutations append audit entries; volume file writes
   never did. Framework changes are exactly the mutations that should be
   audited.
5. **Exact reconcile triggering** replaces fs-watch + debounce for the doc that
   changes least often but matters most.

**Costs, accepted and to be recorded as the §5.2 deviation note:** desired
state now has two homes (files for services, store for framework).
Inspectability moves from `cat framework.json` to `wanfwctl framework show`.
tier1 can never grow a framework-editing UI without a privileged channel — by
design; if ever wanted, ADR-4's approval-gated plan flow is the path, so the
door is gated, not welded shut.

**Implementation checklist:**
1. `StateStore.getFrameworkDoc()` / `setFrameworkDoc()` (meta table).
2. Admin socket `GET/PUT /framework`: validate (same validators as the loader),
   audit (`framework.set`), `onFrameworkChange` callback dep → reconcile.
3. LOAD stage: framework from store (store is already a LOAD dep since T5.1's
   ipam sync); `wanfw_desired` supplies services/plugin-configs only; warn on a
   stale `framework.json` file.
4. Status socket `GET /framework` mirror + allowlist test update. The mirror
   should carry the doc **plus the init-captured operator artifacts** (chosen
   provider's `operatorInstructions`, detected WAN IP) — T5.5's setup page
   renders these, and they belong with the doc in the store.
5. `wanfwctl framework show` / `framework set` (set reads the JSON from stdin,
   which is also what the e2e scripts want).
6. Update every script that writes `framework.json` via busybox volume mounts.

## Decision 2: tier1 first-run password page, gated by a wizard-issued one-time setup token

**Decision.** The admin password is collected, argon2id-hashed, and stored
entirely inside tier1 via a first-run `/setup` page that works only while no
hash exists in `wanfw_tier1state` (the plan's own lean) — **hardened with a
setup token**: `wanfwctl init` generates a random 128-bit token, writes
`{token, createdAt}` to `wanfw_status/setup-token.json` (orchestrator→tier1 is
the legitimate rw→ro direction for that volume), and prints
`open http://<host-lan-ip>:8443/setup and enter token XXXX-XXXX` as its final
step. The setup page requires **both** (no password hash exists) **and** (token
matches, within ~24h of `createdAt`). Once a hash exists the page is dead
regardless of token. No password or hash ever transits a shared volume, the
orchestrator, or the terminal.

**Why:**

1. **The plan's other option is impossible as written.** "A one-time bootstrap
   file in `wanfw_status` that tier1 consumes and deletes at first boot"
   founders on the verified mount table above: tier1 mounts `wanfw_status`
   read-only and *cannot delete anything there*. Making that mount rw would
   break the one-way orchestrator→tier1 status flow. Worse, the credential
   material (plaintext, or an offline-crackable argon2id hash) would ferry
   through a shared volume and — because of that same ro mount — **persist
   there indefinitely** with nothing able to clean it up.
2. **The bare first-run page has a first-boot race.** Anyone on the LAN who
   reaches tier1:8443 before the operator does can set the password and own
   the UI. Modest threat on a home LAN, but this is a security-marketed
   framework and the fix is ~40 lines. The token proves "person running init
   on the host," which is precisely the authority that should own the admin
   account.
3. **Credential handling stays at the edge where it's used.** The wizard never
   touches the password — no plaintext in orchestrator process memory, exec
   plumbing, or terminal scrollback. (This session's own tooling flagged
   agent-side credential generation as out of bounds; the same instinct applies
   to the product: the CLI path shouldn't collect passwords it doesn't need.)

**Details and edges:**
- The token is not a secret after use: once the hash exists, a stale
  `setup-token.json` is inert. Rerunning `wanfwctl init` (or a small
  `wanfwctl setup-token` subcommand, optional nicety) regenerates it.
- Boots where the wizard never ran: the `/setup` page, finding no token file,
  instructs the operator to run `wanfwctl init` on the host. Deliberate — it
  proves host control, and the wizard is *the* onboarding path per §1.2.
- Residual: host root can read the token from the volume. Host root already
  owns everything (ADR-6's own threat model), so this grants no new authority.
- `scripts/seed-admin.mjs` remains the dev-only bootstrap; the token flow is
  the product path.
- T5.3's "Done when" ("fresh-stack init reaches a serving proxy in one
  sitting") is unaffected: proxy serving doesn't depend on tier1 auth. The
  wizard ends by printing the setup URL + token as the final required step, and
  the password-set may happen after the wizard exits.

## One adjacent note for the implementation pass

The plan says the wizard batch-trusts "the six built-ins." There are now seven
in the pluginhost image: the six production plugins (deploy-docker,
network-bridge, network-macvlan, proxy-caddy, dns-namecheap,
cert-letsencrypt-dns01) plus `dns-mock`, which is T4.7 Pebble-test
infrastructure only. The wizard must trust an explicit production list — never
`--builtin-all` — so `dns-mock` is not trusted on real deployments.
